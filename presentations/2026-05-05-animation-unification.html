The Problem

3 systems that conceptually do the same thing: "animate a layer"

Custom

Named Multi-Property

{
  property: 'easeFadeAndPosition-1',
  startTime: 0,
  params: [200, 0.6, {}],
  type: 'in'
}

Expanded by translateCustomAnimationIntoAnimations() into flat tweens

Tween

Direct Property

{
  property: 'opacity',
  startTime: 0.5,
  params: [1, 2, 'easeInCubic'],
  type: 'in'
}

Dot-notation traversal on layer, call with params

Signal

Programmatic

{
  startTime: 0,
  duration: 0.8,
  signalAnimation: {
    name: 'createTypewriterChars',
    options: { fade: true }
  }
}

Factory from registry, reactive signal bindings

Core Insight: In practice, even raw tweens use curated/tested combinations. Arbitrary params are a footgun. Not all easing+distance combos look good.

Motion Canvas key concept: node.property(value, duration, easing) returns a ThreadGenerator. This IS the tween. The framework interpolates frame-by-frame. Signals create reactive bindings that update every frame.

Strategy 1: Factory Functions

Every animation becomes a factory returning { generator: ThreadGenerator }

Core idea: One registry, one interface, one code path. Signal animations already use this exact pattern. Make everything else conform.

Unified Template API

_animations: [
  { startTime: 0.3, duration: 0.8, animation: 'easeFadeAndPosition1', type: 'in' },
  { startTime: 0,   duration: 2.0, animation: 'createTypewriterChars', type: 'in', options: { fade: true } },
  { startTime: 1,   duration: 0.5, animation: 'fadeInCubic', type: 'out' },
]

Unified Executor

for (const anim of layer._animations) {
  const factory = animationRegistry[anim.animation];
  const { generator } = factory({
    targetNode: layer,
    duration: anim.duration,
    type: anim.type,
    ...anim.options,
  });
  generators.push(delay(globalStart + anim.startTime, generator));
}
What disappears: translateCustomAnimationIntoAnimations(), the if/else branching in animateLayers, the distinction between "is this a property path or preset name?"

Strategy 1: How Factories Look

Each animation type becomes the same shape

SIGNAL ANIMATION (unchanged)

// Already a factory today
createTypewriterChars: (opts) => {
  const progress = createSignal(0);
  // bind reactive per-char opacity
  // tween progress 0 -> 1
  return { generator: gen() };
}

SINGLE TWEEN (becomes factory)

// Was: { property: 'opacity', params: [1, 2, ease] }
fadeInCubic: (opts) => ({
  generator: opts.targetNode
    .opacity(1, opts.duration, easeInCubic)
})

CUSTOM ANIMATION (becomes factory)

// Was: { property: 'easeFadeAndPosition-1', params: [...] }
easeFadeAndPosition1: (opts) => {
  function* gen(): ThreadGenerator {
    const baseX = opts.targetNode.position().x;
    const distance = opts.distance ?? 200;

    // Set initial values
    opts.targetNode.position.x(baseX - distance);
    opts.targetNode.opacity(0);

    // Tween to final
    yield* all(
      opts.targetNode.position.x(baseX, opts.duration, easeInCubic),
      opts.targetNode.opacity(1, opts.duration, easeInQuint),
    );
  }
  return { generator: gen() };
}
Pattern: Every factory reads base values at execute-time (after delay), applies initial state, tweens to final state. Self-contained.

Strategy 1: Cons

Con 1: Relative-value logic duplicated per factory

Today translateCustomAnimationIntoAnimations() handles "from = current + offset" generically. With factories, each one writes its own version.

TODAY (generic, one place)

// translateCustomAnimation handles ALL presets
animation.params[0] =
  layer.position().x + fromTo[0];

STRATEGY 1 (per factory)

// Each factory: reads base, applies offset
const baseX = opts.targetNode.position().x;
opts.targetNode.position.x(baseX - distance);
// 30 factories doing the same thing

Risk: Bug in one factory (forget to read base value) doesn't show in others. Inconsistency creeps in.

Con 2: dependsOn easing-inheritance breaks

Current custom animations can declare "opacity inherits easing from position". This is declarative. In a factory, it becomes imperative wiring.

TODAY (declarative)

properties: {
  position: { easing: 'easeInQuint' },
  opacity: { easing: 'dependsOn:position' }
  // ^ auto-resolved
}

STRATEGY 1 (manual sync)

const posEasing = easeInQuint;
yield* all(
  node.position.x(0, dur, posEasing),
  node.opacity(1, dur, posEasing),
  // ^ manually keep in sync
);

Risk: Change easing in one place, forget the other. No compiler warning.

Mitigation: Extract a buildRelativeTween(node, property, offset, duration, easing) helper. Factories call it instead of duplicating logic. Not perfect but reduces risk.

Strategy C: Effect-First with Variants

Everything is an "effect". Named variants = curated option sets.

Core idea: Single flat registry. Each effect has a schema (typed defaults). Variants are frozen option sets for common combos. Overrides for anything custom.

Template API

_animations: [
  // Use a named variant (curated defaults)
  { effect: 'fadeSlide.snappy', startTime: 0, duration: 0.6, type: 'in' },

  // Use base effect with custom params
  { effect: 'fadeSlide', startTime: 0, duration: 0.6, type: 'in', distance: 400, easing: 'easeInExpo' },

  // Signal animation (same interface)
  { effect: 'typewriterChars', startTime: 0.2, duration: 0.8, type: 'in', fade: true },

  // Simple fade (was a raw tween)
  { effect: 'fade', startTime: 3, duration: 0.4, type: 'out' },
]

Variant System

// 50+ numbered presets collapse into:
const variants = {
  'fadeSlide.gentle': {
    distance: 200,
    easing: 'easeInCubic',
    fadeEasing: 'easeInQuint'
  },
  'fadeSlide.snappy': {
    distance: 200,
    easing: 'easeInQuint',
    fadeEasing: 'easeInQuint'
  },
  'fadeSlide.dramatic': {
    distance: 400,
    easing: 'easeInExpo',
    fadeEasing: 'easeInExpo'
  },
};

What this achieves:

  • ~8-10 core effects instead of 50+ numbered presets
  • Each variant is a tested, curated combo
  • Flat named options (no positional params[])
  • Schema validation on registration
  • Still allows overrides when needed

Strategy C: Registry Design

interface EffectRegistration {
  name: string;
  group: string;
  schema: EffectSchema;        // typed defaults + allowed overrides
  create: (ctx: EffectContext) => ThreadGenerator;
}

// Simple tween effect
registerEffect({
  name: 'fade',
  group: 'basic',
  schema: { from: { default: 0 }, to: { default: 1 }, easing: { default: 'easeInCubic' } },
  create: (ctx) => ctx.targetNode.opacity(ctx.to, ctx.duration, ctx.easing),
});

// Multi-property composition
registerEffect({
  name: 'fadeSlide',
  group: 'composed',
  schema: { distance: { default: 200 }, easing: { default: 'easeInCubic' }, fadeEasing: { default: 'easeInQuint' } },
  create: function*(ctx) {
    yield* all(
      ctx.targetNode.position.x(0, ctx.duration, ctx.easing),
      ctx.targetNode.opacity(1, ctx.duration, ctx.fadeEasing),
    );
  },
});

// Signal animation (same interface!)
registerEffect({
  name: 'typewriterChars',
  group: 'text',
  schema: { fade: { default: false } },
  create: (ctx) => createTypewriterChars({
    targetNode: ctx.targetNode,
    duration: ctx.duration,
    type: ctx.type,
    fade: ctx.fade,
  }).generator,
});
Key win: All three types register with the same interface. The executor doesn't know or care what's inside create().

Strategy C: Cons

Con 1: Variant Naming Problem

Who decides what "snappy" vs "gentle" means? What happens when you need something in between?

// Starts clean:
'fadeSlide.gentle':   { distance: 200, easing: 'easeInCubic' }
'fadeSlide.snappy':   { distance: 200, easing: 'easeInQuint' }
'fadeSlide.dramatic': { distance: 400, easing: 'easeInExpo' }

// Then you need: distance 300, easeInQuart
'fadeSlide.medium'?          // vague
'fadeSlide.snappy-wide'?     // naming hell

// Or just override:
{ effect: 'fadeSlide', distance: 300, easing: 'easeInQuart' }
// ^ but then why do variants exist?

Risk: Variants either proliferate (naming hell) or get ignored (everyone uses overrides). Either outcome defeats their purpose.

Con 2: Schema = False Safety

Schema validates types and ranges, but can't validate "looks good on screen".

registerEffect({
  name: 'slideX',
  schema: {
    distance: { type: 'number', default: 1890, min: 50, max: 2000 },
    easing: { type: 'easing', default: 'easeInCubic' },
  },
});

// distance: 1500 passes validation...
// but in a 1080p template, 1500px slide looks wrong visually
// Schema says "valid". Eyes say "broken".

Risk: "Validated" creates false confidence. Real quality still requires visual testing.

Don't unify. Clean up what's broken. Keep what works.

Any animation-system-level feature must be implemented twice.

// Want to build: animation preview tool
if (anim.signalAnimation) {
  // Signal: it's a black-box function
  // Can't inspect params declaratively
  // Can't preview without running it
} else {
  // Tween: can read property, target, duration
  // Easy to build timeline visualization
}
// ^ every new tool hits this fork

Con 2: Zero Discoverability

No way to ask "what animations are available?"

// New template: "I want fade + slide left"
// Where do you look?
// → Open other templates, find similar patterns
// → Copy-paste params manually
// → No autocomplete, no list, no docs

// With a registry:
registry.list()
// → ['fade', 'slideX', 'fadeSlide', 'typewriterChars', ...]
// → each has schema: what options, what defaults

The Parameterization Paradox

This challenge applies to BOTH Strategy 1 and Strategy C

The question: If presets/effects accept arbitrary distance, duration, easing, direction as parameters... how is that different from raw tweens? You've just renamed the system.

Side by Side

STRATEGY C

{
  effect: 'slideX',
  startTime: 0,
  duration: 0.6 * (pacing + pacingStep),
  distance: -1080,
  easing: 'easeOutQuint',
  type: 'in'
}

CURRENT RAW TWEEN

{
  property: 'position.x',
  startTime: 0,
  params: [0, 0.6 * (pacing + pacingStep),
           'easeOutQuint'],
  type: 'in',
  initialValue: -1080
}

What Unification Actually Gives You (if not constraint)

Gets

  • Defaults: don't specify what's common
  • Discoverability: list available animations
  • One API: same shape for everything
  • Tooling: build preview/timeline tools
  • Naming: intent over implementation

Does NOT Get

  • Constraint: can't block arbitrary values without breaking use cases
  • Safety: schema can't validate visual quality
  • Fewer params: same info still needed
  • Simpler mental model: still need to know distance, easing, duration
Bottomline: Unification's real value = defaults + discoverability + unified tooling. NOT constraint or safety.

Critical Finding: Dead Code

The 48 custom animation presets have zero consumers

No template uses the custom animation presets. 100% of templates use raw property tweens directly. The system being proposed as the unification target... was never adopted.

Decision Framework

The answer depends on WHAT problem you're actually solving

"I want AI/tooling to generate templates"

→ Strategy C (constrained, schema-validated, discoverable effects)

AI needs a bounded vocabulary. Schema = structured generation target.

"I want one code path and clean architecture"

→ Strategy 1 (unified factory registry, Motion Canvas native)

Simplest technically. Skip variants, just make everything a factory.

"Nothing is broken, just messy"

→ 80/20 (delete dead presets, add validation, keep raw tweens)

Pragmatic. Zero risk. Spend time on features instead.

"I want reusability + building a library of tested animations"

→ Strategy 1 or C (registry = library you grow over time)

Either works. C adds variant curation on top. 1 is simpler to start.

Key question: Do you plan to have tooling (AI, visual editor, animation picker) that needs to enumerate available animations? If yes, you need a registry (Strategy 1 or C). If no, 80/20 might be enough.

What's Actually at Stake

Separating real value from theoretical purity

Real Value of Unification

  • Defaults: new template starts with sane values, only override what's different
  • Discoverability: "what animations exist?" has an answer
  • One mental model: everything is an "effect" with a name
  • Tooling foundation: preview tools, timeline editors, AI generation
  • Consistency: same animation looks the same across templates
  • Onboarding: new template = pick from menu, not copy-paste from other templates

NOT Achieved by Unification

  • Constraint: can't block bad values without losing flexibility
  • Visual quality: schema validation != looks good
  • Fewer decisions: still need to choose distance, easing, duration
  • Bug prevention: current system doesn't have animation bugs
  • Speed: copy-paste from another template is fast enough today
SO
The honest trade: Unification is an investment in future tooling and consistency. It doesn't fix a current bug. Its value scales with: (a) number of templates you'll create, (b) whether non-human authors (AI) will create animations, (c) whether you'll build animation tooling.

Recommended Path Forward

If unifying: Strategy 1 (Factory Functions)

Simpler than C, avoids variant naming problem, Motion Canvas native idiom

Step 1: Foundation

  • Create animationRegistry (Map<string, factory>)
  • Register existing signal animations (already factories)
  • Add compat shim: old format falls through to current logic

Step 2: Core Effects

  • Convert top 5-8 most-used patterns into named factories
  • Extract buildRelativeTween() helper to avoid duplication
  • New templates use animation: 'name' format

Step 3: Grow Organically

  • Add factories as you create new templates
  • Old templates don't need rewriting (compat shim)
  • Delete dead customAnimations array
Key principle: Don't pre-build 50 factories. Start with the registry + compat shim + 5 core effects. Let the library grow from real needs.
If NOT unifying: Delete the 48 dead presets. Add runtime validation (easing string exists, duration > 0). Keep raw tweens + signal animations as-is. Move on.
